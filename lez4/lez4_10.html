Build, linking, ODR e dipendenze

Obiettivo della pagina

Chiudere il modello di compilazione necessario per lavorare bene con classi distribuite su piu file.

Idea chiave: compilazione e linking sono fasi diverse; molti errori su header e duplicazioni nascono proprio dal confondere queste due fasi.

Compilare non e linkare

  • compilazione: ogni translation unit viene trasformata in un file oggetto;
  • linking: i file oggetto vengono uniti in un solo eseguibile.

Un progetto puo compilare bene file per file e poi fallire al link se le definizioni non sono coerenti.

ODR: One Definition Rule

In forma semplificata, una funzione non inline o un oggetto con linkage esterno non devono essere definiti piu volte nel programma.

Se definisci la stessa funzione libera in due punti diversi, il linker trova due versioni concorrenti e protesta.

Dipendenze minime

  • includi solo cio che serve davvero nell'header;
  • usa forward declaration quando basta conoscere un nome di tipo;
  • evita using namespace negli header;
  • sposta nel .cpp cio che appartiene solo all'implementazione.

Messaggio pratico

Molte scelte "di stile" su header e file sorgente in realta sono scelte di correttezza e manutenibilita.

Il progetto cresce bene solo se i confini tra i file restano puliti come i confini tra le classi.

Errore tipico: correggere un errore di dipendenze aggiungendo include ovunque "finche compila". Spesso cosi nascondi il problema invece di risolverlo.
