Overload e overload const/non-const

Obiettivo della pagina

Capire quando ha senso offrire piu varianti dello stesso nome e come cambia l'interfaccia tra oggetti const e non const.

Idea chiave: l'overload e utile quando il significato dell'operazione resta lo stesso, ma cambiano il contesto o i parametri.

Overload per parametri

Puoi usare lo stesso nome per operazioni concettualmente uguali ma con firme diverse.

class Logger {
public:
    void write(const std::string& s);
    void write(int value);
};

Qui il significato resta "scrivere qualcosa", ma il tipo dell'argomento cambia.

Overload const/non-const

Una variante comune e offrire due versioni dello stesso accesso: una per sola lettura e una che permette modifica.

class Buffer {
private:
    int data_[10];
public:
    int& at(int i) { return data_[i]; }
    const int& at(int i) const { return data_[i]; }
};

Se l'oggetto e const, il compilatore sceglie la versione const. Questo rafforza il contratto del tipo.

Quando l'overload migliora davvero il design

  • quando il nome rappresenta una sola idea concettuale;
  • quando la scelta tra le varianti e naturale per chi usa il tipo;
  • quando eviti nomi artificiali come getXConst o writeInt se il significato e gia chiaro dal contesto.
Errore tipico: usare lo stesso nome per operazioni che sembrano simili ma hanno effetti molto diversi. In quel caso l'overload confonde invece di chiarire.
