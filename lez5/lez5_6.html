Usare bene il polimorfismo
override, distruttore virtuale, classi astratte, override vs overload

Che cosa aggiungiamo rispetto alla pagina precedente

Nella pagina precedente abbiamo capito il meccanismo del dispatch dinamico: con virtual il programma usa il tipo reale dell'oggetto. Ma per usare questo meccanismo in modo robusto non basta scrivere una parola chiave e sperare che tutto vada bene.

Qui entrano in gioco quattro idee strettamente collegate:

  • override, per far controllare al compilatore che stiamo davvero ridefinendo il metodo giusto;
  • il distruttore virtuale, per evitare errori gravi quando si distrugge tramite puntatore a base;
  • le classi astratte, per definire contratti comuni senza istanziare direttamente la base;
  • la distinzione tra override e overload, che non vanno confusi.

Perché override è importante

override dice al compilatore:

"Sto intenzionalmente sovrascrivendo un metodo virtuale della base."

Questa piccola parola chiave è molto più preziosa di quanto sembri. Non serve a "rendere più OO" il codice e non cambia il comportamento del programma in senso magico. Serve a far controllare al compilatore che la tua intenzione sia coerente con la firma del metodo nella base.

Se la firma non corrisponde, il compilatore segnala errore. Ed è esattamente ciò che vogliamo: meglio un errore in compilazione che un programma che sembra corretto ma in realtà non sta ridefinendo il metodo che pensavi.

Esempio corretto con override

struct Base {
    virtual void f() const {}
};

struct Der : Base {
    void f() const override {}
};

Qui tutto è coerente. La derivata dichiara esplicitamente di stare ridefinendo f(), e il compilatore conferma che la firma è compatibile con quella della base.

Errore tipico: firma quasi uguale, ma non uguale davvero

Il testo insiste giustamente su un errore molto comune:

struct Base {
    virtual void f() const {}
};

struct Der : Base {
    // void f() {}   // NON è override: manca const
    void f() const override {}
};

Basta dimenticare un const per cambiare la firma. In quel caso non staresti più ridefinendo il metodo virtuale della base: staresti dichiarando un altro metodo. Proprio per questo override è così utile.

Regola pratica: metti override su tutti i metodi virtuali ridefiniti. Non è rumore: è un controllo di correttezza.

Perché serve il distruttore virtuale

Se una classe base viene usata polimorficamente, il suo distruttore deve essere virtuale. Questo non è un raffinamento stilistico: è una regola di sicurezza molto concreta.

Quando distruggi un oggetto tramite un puntatore a base, il programma deve sapere che deve chiamare anche il distruttore della derivata reale. Se il distruttore della base non è virtuale, questa catena può rompersi.

Esempio problematico

#include <iostream>

struct Base {
    ~Base() { std::cout << "~Base\n"; }
};

struct Der : Base {
    ~Der() { std::cout << "~Der\n"; }
};

int main() {
    Base* p = new Der();
    delete p;  // comportamento indefinito
}

Qui il problema non è solo teorico. Eliminando tramite Base* potresti non chiamare il distruttore della derivata. Se la derivata gestisce risorse, il rischio è serio: memoria non rilasciata, file lasciati aperti, stato incoerente.

Regola pratica: se una classe ha almeno un metodo virtuale, di solito deve avere anche:

virtual ~Base() = default;

Classi astratte e metodi puri

Una classe astratta definisce un contratto comune. Non può essere istanziata direttamente. Questo è esattamente ciò che vogliamo quando la base rappresenta un'idea generale, ma non un oggetto concreto completo.

La sintassi chiave è il metodo virtuale puro:

virtual double area() const = 0;

Quel = 0 significa che la base impone il contratto ma non fornisce un'implementazione completa per quel metodo. Saranno le derivate concrete a farlo.

Esempio: gerarchia Forma

#include <cmath>
#include <iostream>
#include <memory>
#include <vector>

class Forma {
public:
    virtual ~Forma() = default;
    virtual double area() const = 0;

    virtual void stampa() const {
        std::cout << "Area=" << area() << "\n";
    }
};

class Cerchio : public Forma {
public:
    explicit Cerchio(double r) : r_(r) {}

    double area() const override {
        constexpr double pi = 3.141592653589793;
        return pi * r_ * r_;
    }
private:
    double r_;
};

class Rettangolo : public Forma {
public:
    Rettangolo(double w, double h) : w_(w), h_(h) {}

    double area() const override { return w_ * h_; }
private:
    double w_, h_;
};

int main() {
    std::vector<std::unique_ptr<Forma>> v;
    v.push_back(std::make_unique<Cerchio>(2.0));
    v.push_back(std::make_unique<Rettangolo>(3.0, 4.0));

    for (const auto& f : v) {
        f->stampa();
    }
}

Perché qui si usano puntatori e unique_ptr?

Il testo risponde in modo molto chiaro, e va tenuto così:

  • Forma è astratta, quindi non si possono creare oggetti Forma diretti;
  • i puntatori a base permettono il polimorfismo;
  • unique_ptr evita new e delete manuali e gestisce la memoria in modo sicuro.

Questa è una pagina importante perché salda insieme linguaggio e stile moderno: base astratta, polimorfismo e gestione sicura della memoria entrano nello stesso esempio.

override non è overload

Questi due concetti non vanno confusi.

override:

  • ridefinisce un metodo virtuale della base;
  • richiede una firma compatibile.

overload:

  • usa lo stesso nome, ma parametri diversi;
  • non riguarda direttamente il polimorfismo dinamico.

Questa distinzione è fondamentale per non fraintendere il ruolo di override: non stiamo creando "un altro metodo con lo stesso nome", stiamo dicendo che una funzione virtuale già esistente deve essere ridefinita con la firma giusta.

Name hiding e using Base::g;

Il testo chiude questa parte con un caso molto utile:

struct Base {
    void g(int) {}
    void g(double) {}
};

struct Der : Base {
    using Base::g;
    void g(int) {}
};

Senza using Base::g;, la nuova definizione in Der nasconde le altre overload della base. Questo fenomeno si chiama name hiding.

Il punto didattico qui è sottile ma importante: non tutto ciò che riguarda "stesso nome" ha a che fare con il polimorfismo dinamico. A volte il problema è molto più semplice: la derivata ha nascosto overload utili della base.

Sintesi finale

  • override serve a far verificare al compilatore che la ridefinizione sia corretta;
  • un distruttore virtuale è necessario quando la base viene usata polimorficamente;
  • una classe astratta definisce un contratto comune e non può essere istanziata direttamente;
  • std::unique_ptr<Base> è spesso il modo più pulito per gestire oggetti polimorfici;
  • override e overload sono concetti diversi, e il name hiding è un problema a parte.

Sintesi finale: il polimorfismo non si usa bene solo con virtual; si usa bene quando la base è progettata per essere sicura, chiara e controllabile dal compilatore.
