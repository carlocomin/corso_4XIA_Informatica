Polimorfismo: virtual e dispatch dinamico
Tipo statico, tipo dinamico, metodo chiamato davvero

Che cos'è il polimorfismo

Il polimorfismo consente di trattare oggetti di tipi derivati diversi attraverso un riferimento o un puntatore alla classe base, mantenendo però il comportamento corretto del tipo reale.

Questa frase contiene già l'idea decisiva della pagina. Non stiamo parlando solo di "ereditarietà" in senso statico. Stiamo parlando della capacità del programma di usare una base comune senza perdere la specificità dell'oggetto concreto che c'è dietro.

La parola chiave centrale, in C++, è virtual.

Tipo statico e tipo dinamico

Per capire il polimorfismo bisogna distinguere con chiarezza due nozioni:

  • tipo statico: il tipo con cui una variabile, un puntatore o un riferimento è dichiarato nel codice;
  • tipo dinamico: il tipo reale dell'oggetto a cui quella variabile, quel puntatore o quel riferimento si riferiscono.

Se un puntatore ha tipo Animale* ma punta a un oggetto Cane, allora:

  • il tipo statico è Animale*;
  • il tipo dinamico dell'oggetto reale è Cane.

Domanda guida: quando chiami un metodo tramite una base, il programma deve guardare il tipo statico o quello dinamico? La risposta dipende proprio dal fatto che il metodo sia o non sia virtual.

Senza virtual: binding statico

Se un metodo non è dichiarato virtual, il metodo chiamato dipende dal tipo statico della variabile attraverso cui fai la chiamata.

Il testo lo mostra con questo esempio:

#include <iostream>

struct Animale {
    void verso() const { std::cout << "Verso generico\n"; }
};

struct Cane : public Animale {
    void verso() const { std::cout << "Bau!\n"; }
};

int main() {
    Cane c;
    Animale* p = &c;
    p->verso();
}

Output del caso senza virtual

L'output è:

Verso generico

A prima vista può sembrare strano, perché l'oggetto reale è un Cane. Ma il compilatore guarda il tipo statico del puntatore, che è Animale*, e dato che il metodo non è virtuale decide la chiamata in modo statico.

Perché accade? Perché p ha tipo Animale* e il metodo non è virtuale. Il compilatore sceglie la versione da chiamare guardando il tipo dichiarato, non il tipo reale dell'oggetto.

Con virtual: dispatch dinamico

Se invece il metodo è dichiarato virtual nella base, il metodo chiamato dipende dal tipo dinamico dell'oggetto reale. Qui il comportamento cambia radicalmente, ed è proprio questo il cuore del polimorfismo.

Il testo propone questo secondo esempio:

#include <iostream>

struct Animale {
    virtual void verso() const { std::cout << "Verso generico\n"; }
    virtual ~Animale() = default;
};

struct Cane : public Animale {
    void verso() const override { std::cout << "Bau!\n"; }
};

struct Gatto : public Animale {
    void verso() const override { std::cout << "Miao!\n"; }
};

void faiVerso(const Animale& a) {
    a.verso();
}

int main() {
    Cane c;
    Gatto g;
    faiVerso(c);
    faiVerso(g);
}

Output del caso con virtual

L'output ora diventa:

Bau!
Miao!

Qui il programma usa il tipo reale dell'oggetto. Anche se faiVerso riceve un riferimento a Animale, viene chiamata la versione corretta del metodo in base all'oggetto concreto passato alla funzione.

Idea chiave: il polimorfismo non significa solo "ho classi in ereditarietà". Significa che il codice può lavorare con la base e ottenere comunque il comportamento corretto della derivata reale.

Perché l'esempio con faiVerso(const Animale&) è così importante

Questa piccola funzione è didatticamente perfetta:

void faiVerso(const Animale& a) {
    a.verso();
}

Mostra che il codice chiamante non ha bisogno di sapere se sta ricevendo un Cane, un Gatto o un altro animale. Gli basta sapere che sta lavorando con qualcosa che rispetta il contratto di Animale.

Questa è la vera forza del polimorfismo: il codice ad alto livello si semplifica, perché smette di fare controlli sul tipo concreto e si affida a un'interfaccia comune ben progettata.

Confronto rapido: senza e con virtual

Situazione Da che cosa dipende il metodo chiamato? Effetto pratico
Senza virtual Dal tipo statico della variabile Si ottiene binding statico
Con virtual Dal tipo dinamico dell'oggetto reale Si ottiene dispatch dinamico

Questa tabella condensa l'intero passaggio teorico della pagina. Il resto della lezione servirà a rendere questo meccanismo robusto, leggibile e progettualmente corretto, introducendo override, distruttore virtuale e classi astratte.

Sintesi finale

  • il polimorfismo permette di usare una base comune senza perdere il comportamento corretto del tipo reale;
  • senza virtual, il metodo chiamato dipende dal tipo statico;
  • con virtual, il metodo chiamato dipende dal tipo dinamico dell'oggetto;
  • riferimenti e puntatori a base sono il canale naturale attraverso cui il polimorfismo opera;
  • una funzione come faiVerso(const Animale&) mostra perché il polimorfismo semplifica il codice cliente.

Sintesi finale: virtual dice al programma di non fermarsi al tipo con cui una variabile è dichiarata, ma di guardare il tipo reale dell'oggetto per scegliere il comportamento giusto.
