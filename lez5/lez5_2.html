Sintassi base della derivazione
public inheritance, private, protected, public

Da dove partiamo

Nella pagina precedente abbiamo chiarito il criterio concettuale: usare ereditarietà solo quando la relazione tra derivata e base è davvero un "è-un". Ora possiamo vedere come questa idea prende forma in C++.

Questa pagina copre la sintassi essenziale e i livelli di accesso. Non stiamo ancora parlando di polimorfismo dinamico: per adesso ci basta capire che cosa eredita davvero una classe derivata, che cosa può usare, e che cosa invece resta nascosto.

Idea chiave: la derivazione public è la forma normale quando vogliamo modellare una vera relazione concettuale tra base e derivata.

Derivazione public: la forma più comune

La forma più comune e più importante da padroneggiare è la derivazione public. Quando scriviamo una classe derivata in questo modo, stiamo dicendo che l'interfaccia pubblica della base resta pubblica anche nella derivata.

In altri termini, se la base espone operazioni che hanno senso per tutti i suoi casi particolari, allora anche la derivata le eredita come parte della propria interfaccia pubblica. Questo è coerente con il significato di IS-A: se Studente è una Persona, allora deve poter fare almeno le cose che ogni persona sa fare, oltre ad eventuali comportamenti specifici.

Esempio completo: Persona e Studente

Qui sotto c'è l'esempio centrale del testo sorgente. Va letto lentamente, perché in poche righe mostra già quasi tutto quello che ci serve: costruttore della base, derivazione public, uso di un metodo protected, attributo private e metodo ereditato.

#include <iostream>
#include <string>
#include <utility>

class Persona {
public:
    Persona(std::string nome) : nome_(std::move(nome)) {}

    void saluta() const {
        std::cout << "Ciao, sono " << nome_ << "\n";
    }

protected:
    const std::string& nome() const { return nome_; }

private:
    std::string nome_;
};

class Studente : public Persona {
public:
    Studente(std::string nome, int classe)
        : Persona(std::move(nome)), classe_(classe) {}

    void presenta() const {
        std::cout << "Sono " << nome() << " e sono in " << classe_ << "a.\n";
    }

private:
    int classe_;
};

int main() {
    Studente s("Carlo", 4);
    s.saluta();
    s.presenta();
}

Output atteso

Il programma produce:

Ciao, sono Carlo
Sono Carlo e sono in 4a.

Questo output è importante perché mostra due cose diverse. La prima riga arriva da un metodo ereditato dalla base. La seconda riga arriva da un metodo specifico della derivata. In un solo main() vediamo quindi convivere riuso e specializzazione.

Che cosa osservare davvero nel codice

Il testo sorgente evidenzia tre osservazioni che vanno fissate bene, perché sono la base di quasi tutte le scelte successive.

  • Studente eredita saluta() da Persona. Questo è l'effetto più visibile della derivazione public: un metodo pubblico della base resta utilizzabile anche sulla derivata.
  • Studente non può accedere direttamente a nome_. L'attributo è private dentro Persona, quindi la derivata non lo vede direttamente. Questo dettaglio è fondamentale: derivare non significa "avere accesso a tutto".
  • Studente può usare nome(). Il metodo è dichiarato protected, quindi è accessibile nelle classi derivate ma non dal codice esterno.

Tradotto in termini progettuali: la base protegge i propri dati interni, ma può offrire alle derivate alcuni punti di accesso controllati se questo aiuta a specializzare il comportamento senza rompere l'incapsulamento.

Il ruolo di private, protected e public nella base

Quando una classe è destinata a fare da base, i tre livelli di accesso non vanno pensati solo in rapporto all'esterno, ma anche in rapporto alle future derivate.

Specifica nella base Che cosa significa per la derivata Che cosa significa per il codice esterno
private Non accessibile direttamente Non accessibile
protected Accessibile nella derivata Non accessibile
public Resta accessibile normalmente con derivazione public Accessibile

Questa tabella riassume una regola semplice ma decisiva:

  • private nella base: non accessibile dalla derivata;
  • protected nella base: accessibile nella derivata, ma non dall'esterno;
  • public nella base: resta accessibile normalmente con derivazione public.

Perché nome_ è private ma nome() è protected?

Questa scelta è didatticamente molto istruttiva. Se la classe base rendesse l'attributo nome_ direttamente accessibile alla derivata, la derivata conoscerebbe troppo dei dettagli interni della base. Invece, esponendo un metodo protected come nome(), la base conserva il controllo del proprio stato ma offre alle derivate una vista controllata sul dato che serve davvero.

In altre parole, protected può essere utile quando vogliamo aiutare le classi derivate a lavorare con la base senza aprire completamente l'interno della classe. Va comunque usato con attenzione: una base troppo esposta alle derivate rischia di diventare fragile nel tempo.

Il costruttore della derivata e il costruttore della base

Guarda questa riga della classe Studente:

Studente(std::string nome, int classe)
    : Persona(std::move(nome)), classe_(classe) {}

Qui la derivata non "si costruisce da sola". Deve inizializzare anche la parte di base. Il costruttore di Studente richiama esplicitamente il costruttore di Persona nella lista di inizializzazione. Questo punto diventerà ancora più importante nella prossima pagina, quando parleremo di ordine di costruzione e distruzione.

Per ora basta fissare il principio: un oggetto derivato contiene anche la parte base, quindi la base deve essere inizializzata correttamente fin dall'inizio.

Esistono anche derivazione protected e private

Il linguaggio C++ prevede anche altre forme di derivazione:

  • protected inheritance;
  • private inheritance.

In questi casi cambia la visibilità dei membri ereditati nella derivata. Non entrare troppo presto nei dettagli è una scelta saggia: all'inizio, la cosa davvero importante è capire bene la derivazione public, perché è quella che modella in modo naturale una vera relazione "è-un".

Negli usi introduttivi e in molti progetti applicativi:

  • si usa soprattutto public inheritance per modellare una vera relazione concettuale tra tipi;
  • negli altri casi si preferisce spesso la composizione.

Regola pratica: se stai pensando a private inheritance solo per riusare implementazione, fermati un attimo e chiediti se non sia più pulita la composizione.

Sintesi finale

Se questa seconda lezione è stata capita bene, dovresti aver fissato questi punti:

  • la derivazione public è la forma standard quando la derivata è davvero un caso particolare della base;
  • non tutto ciò che sta nella base diventa accessibile alla derivata: private resta interno alla base;
  • protected può essere un ponte controllato verso le derivate;
  • la derivata può riusare l'interfaccia pubblica della base e aggiungere comportamento specifico;
  • il costruttore della derivata deve anche inizializzare la parte di base.

Sintesi finale: la derivazione public non è solo una formula sintattica. È il modo con cui C++ rende concreta, nel codice, una gerarchia progettata in modo sensato.

Passo successivo. Ora che la sintassi di base è chiara, possiamo affrontare uno dei punti in cui si sbaglia più facilmente: il ciclo di vita degli oggetti in gerarchia. Nella Lez5_3 vedremo ordine di costruzione, ordine di distruzione e inizializzazione della base.
