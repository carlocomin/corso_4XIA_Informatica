Esercizio pratico finale
Gerarchia Dipendente, polimorfismo, smart pointer

Obiettivo dell'esercizio pratico

Qui la teoria deve diventare progetto. Non basta più dire che una classe è astratta o che un metodo è virtuale: bisogna costruire una piccola gerarchia credibile, usarla in modo polimorfico e far vedere che le scelte progettuali reggono anche nel main().

L'esercizio è molto buono perché costringe a tenere insieme più concetti nello stesso programma: base astratta, override, contenitore di oggetti eterogenei, assenza di slicing, distruttore virtuale, e scelta sensata tra puntatori grezzi e smart pointer.

La sezione pratica dell'esercitazione ti chiede proprio questo: passare da domande e definizioni a un piccolo progetto funzionante, in cui il polimorfismo non sia nominato soltanto in teoria ma usato davvero.

Idea chiave: il polimorfismo diventa davvero utile quando il codice chiamante lavora con un'interfaccia comune e non ha bisogno di conoscere tutti i dettagli delle classi concrete.

Testo dell'esercizio

Implementa una piccola gerarchia di classi per la gestione degli stipendi.

Requisiti richiesti:

  1. Parti dalla classe astratta Dipendente e inserisci:
    • distruttore virtuale;
    • metodo puro double stipendioMensile() const = 0;;
    • metodo virtuale void stampa() const.
  2. Definisci poi due classi derivate:
    • DipendenteFisso (attributo: stipendio fisso);
    • DipendenteOrario (attributi: pagaOraria, oreMese).
  3. In entrambe usa override dove serve.
  4. Scrivi quindi la funzione:
    double totaleStipendi(const std::vector<Dipendente*>& v);
  5. Infine, nel main, crea almeno 4 dipendenti di tipo misto, stampa lo stipendio di ciascuno e calcola il totale complessivo.

Vincoli:

  • evita slicing;
  • non usare new/delete manuali;
  • usa nomi significativi e codice ordinato.

Strategia progettuale prima di scrivere codice

La parte più interessante dell'esercizio è che contiene una piccola tensione progettuale voluta. Da un lato la consegna chiede una funzione che riceve std::vector<Dipendente*>. Dall'altro vieta new e delete manuali. La soluzione migliore non è ignorare uno dei due vincoli, ma farli convivere in modo pulito.

Nel main() possiamo gestire davvero gli oggetti con std::unique_ptr<Dipendente>, evitando quindi gestione manuale della memoria. Quando dobbiamo chiamare totaleStipendi(...), possiamo costruire una piccola "vista" temporanea di puntatori non proprietari, ricavati dagli unique_ptr con get(). In questo modo:

  • la ownership resta chiara e sicura;
  • la funzione rispetta la firma chiesta dalla consegna;
  • non c'è slicing, perché lavoriamo sempre tramite puntatori alla base;
  • non usiamo mai delete a mano.

Messaggio metodologico: anche quando una consegna usa puntatori grezzi, in C++ moderno conviene chiedersi chi possiede davvero gli oggetti. Qui la risposta giusta è il contenitore di unique_ptr.

Che cosa osservare nella soluzione

  • Base astratta: Dipendente non si può istanziare direttamente, ma definisce il contratto comune.
  • Metodo virtuale puro: ogni derivata deve spiegare come si calcola il proprio stipendio.
  • Metodo virtuale concreto: stampa() può avere una versione di default nella base, ma le derivate possono personalizzarla.
  • Distruttore virtuale: è corretto e necessario per una base usata polimorficamente.
  • Polimorfismo vero: il totale viene calcolato senza chiedere "che tipo sei?", ma chiamando lo stesso metodo su oggetti diversi.

Soluzione guidata

Prima prova a progettare la gerarchia da solo. Poi apri la soluzione e verifica soprattutto il perché delle scelte, non solo la sintassi.

#include <iostream>
#include <memory>
#include <string>
#include <utility>
#include <vector>

class Dipendente {
public:
    virtual ~Dipendente() = default;

    virtual double stipendioMensile() const = 0;

    virtual void stampa() const {
        std::cout << "Dipendente: stipendio = "
                  << stipendioMensile()
                  << " euro\n";
    }
};

class DipendenteFisso : public Dipendente {
private:
    std::string nome_;
    double stipendioFisso_;

public:
    DipendenteFisso(std::string nome, double stipendioFisso)
        : nome_{std::move(nome)},
          stipendioFisso_{stipendioFisso >= 0.0 ? stipendioFisso : 0.0} {
    }

    double stipendioMensile() const override {
        return stipendioFisso_;
    }

    void stampa() const override {
        std::cout << "[Fisso] " << nome_
                  << " - stipendio = "
                  << stipendioMensile()
                  << " euro\n";
    }
};

class DipendenteOrario : public Dipendente {
private:
    std::string nome_;
    double pagaOraria_;
    double oreMese_;

public:
    DipendenteOrario(std::string nome, double pagaOraria, double oreMese)
        : nome_{std::move(nome)},
          pagaOraria_{pagaOraria >= 0.0 ? pagaOraria : 0.0},
          oreMese_{oreMese >= 0.0 ? oreMese : 0.0} {
    }

    double stipendioMensile() const override {
        return pagaOraria_ * oreMese_;
    }

    void stampa() const override {
        std::cout << "[Orario] " << nome_
                  << " - paga oraria = " << pagaOraria_
                  << ", ore mese = " << oreMese_
                  << ", stipendio = " << stipendioMensile()
                  << " euro\n";
    }
};

double totaleStipendi(const std::vector<Dipendente*>& v) {
    double totale = 0.0;

    for (const Dipendente* d : v) {
        if (d != nullptr) {
            totale += d->stipendioMensile();
        }
    }

    return totale;
}

int main() {
    std::vector<std::unique_ptr<Dipendente>> personale;

    personale.push_back(std::make_unique<DipendenteFisso>("Anna", 1850.0));
    personale.push_back(std::make_unique<DipendenteOrario>("Luca", 12.5, 120.0));
    personale.push_back(std::make_unique<DipendenteFisso>("Marta", 2100.0));
    personale.push_back(std::make_unique<DipendenteOrario>("Paolo", 15.0, 95.0));

    std::cout << "Stipendi del personale:\n";
    for (const auto& d : personale) {
        d->stampa();
    }

    std::vector<Dipendente*> vista;
    vista.reserve(personale.size());
    for (const auto& d : personale) {
        vista.push_back(d.get());
    }

    std::cout << "\nTotale stipendi = "
              << totaleStipendi(vista)
              << " euro\n";
}

Perché questa soluzione è buona: il main() possiede davvero gli oggetti tramite unique_ptr, ma la funzione di calcolo vede solo una collezione di Dipendente*. Questa separazione tra ownership e uso polimorfico è pulita e molto istruttiva.

Osserva bene: totaleStipendi(...) non controlla mai se un oggetto è fisso o orario. Chiama semplicemente stipendioMensile(). Questo è il punto esatto in cui il polimorfismo fa risparmiare condizioni, cast e codice duplicato.

Commento ragionato alla soluzione

La base Dipendente esprime il contratto comune: ogni dipendente deve saper restituire il proprio stipendio mensile. La base non dice come si calcola, perché quello dipende dal tipo concreto. Questo è il motivo corretto per usare un metodo virtuale puro.

Le classi derivate non aggiungono semplicemente campi: aggiungono una politica di calcolo. DipendenteFisso restituisce il valore fisso; DipendenteOrario calcola prodotto tra paga oraria e ore mensili. È proprio questa variabilità controllata che rende sensata la gerarchia.

La funzione totaleStipendi(...) è volutamente semplice. Se il progetto è buono, il codice ad alto livello deve diventare quasi banale. Quando una base astratta è progettata bene, la logica di aggregazione spesso si riduce a un ciclo e a una chiamata virtuale.

Bonus: aggiungere DipendenteProvvigione

Se avanza tempo, la consegna chiede una terza derivata con formula:

stipendio = fissoBase + percentuale * venditeMensili

Il punto didattico del bonus non è la formula in sé. È verificare che una nuova derivata possa entrare nel sistema senza costringerti a cambiare la firma di totaleStipendi(...) e senza rompere il ciclo principale del main().

class DipendenteProvvigione : public Dipendente {
private:
    std::string nome_;
    double fissoBase_;
    double percentuale_;
    double venditeMensili_;

public:
    DipendenteProvvigione(std::string nome,
                          double fissoBase,
                          double percentuale,
                          double venditeMensili)
        : nome_{std::move(nome)},
          fissoBase_{fissoBase >= 0.0 ? fissoBase : 0.0},
          percentuale_{percentuale >= 0.0 ? percentuale : 0.0},
          venditeMensili_{venditeMensili >= 0.0 ? venditeMensili : 0.0} {
    }

    double stipendioMensile() const override {
        return fissoBase_ + percentuale_ * venditeMensili_;
    }

    void stampa() const override {
        std::cout << "[Provvigione] " << nome_
                  << " - stipendio = "
                  << stipendioMensile()
                  << " euro\n";
    }
};

Osservazione importante: il resto del programma non deve essere riscritto. Ti basta fare una nuova push_back(std::make_unique<DipendenteProvvigione>(...)) nel contenitore e il calcolo del totale continua a funzionare.

Riflessione finale: quando useresti dynamic_cast?

In questo progetto, quasi mai. La struttura della gerarchia è stata progettata proprio per evitare cast espliciti. Se ti serve solo conoscere o usare il comportamento stipendiale, il metodo virtuale basta e avanza. Fare dynamic_cast per chiedere "sei un DipendenteOrario?" sarebbe il segnale che stai riportando fuori dalla gerarchia una logica che dovrebbe stare dentro le classi.

dynamic_cast può avere senso solo in casi particolari: per esempio se, in una fase di debug o in una funzionalità secondaria, devi riconoscere un tipo concreto per gestire un comportamento davvero eccezionale. Ma nel flusso ordinario del programma, la scelta migliore è quasi sempre affidarsi ai metodi virtuali.

Regola pratica: se il tuo progetto richiede cast continui per lavorare con una gerarchia, è probabile che l'interfaccia della base non stia esprimendo abbastanza bene il contratto comune.

Checklist conclusiva della lezione 5
  • La relazione tra base e derivate è davvero un "è-un"?
  • La base ha un distruttore virtuale se viene usata polimorficamente?
  • I metodi ridefiniti usano override?
  • Stai evitando slicing con riferimenti, puntatori o smart pointer?
  • La logica ad alto livello usa il contratto comune invece di fare cast sui tipi concreti?

Se a tutte queste domande rispondi bene, allora non stai solo scrivendo codice con ereditarietà: stai usando l'ereditarietà in modo progettualmente corretto.
