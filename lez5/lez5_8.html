Ereditarietà multipla: che cos'è e quando ha senso

In C++ una classe può derivare da più classi base. Questo fatto rende il linguaggio molto espressivo, ma anche più delicato da progettare. L'ereditarietà multipla non è "sbagliata" in assoluto: va capita e usata con criterio.

Il testo propone un esempio semplice e ragionevole:

struct Stampabile {
    virtual ~Stampabile() = default;
    virtual void stampa() const = 0;
};

struct Salvabile {
    virtual ~Salvabile() = default;
    virtual void salva() const = 0;
};

struct Documento : Stampabile, Salvabile {
    void stampa() const override {}
    void salva() const override {}
};

Questo uso è spesso sensato quando le basi rappresentano interfacce astratte. In questo caso non stiamo ereditando stato o implementazioni complesse da due lati diversi: stiamo dicendo che Documento soddisfa due contratti distinti.

Diamond problem

Il problema compare quando due classi derivano dalla stessa base e una terza eredita da entrambe. Il disegno classico è questo:

    A
   / \
  B   C
   \ /
    D

Senza ereditarietà virtuale, D può contenere due copie separate di A. Questo crea ambiguità e complicazioni: quale copia di A stai usando? quale dato comune stai leggendo? quale metodo stai invocando?

Per evitare il problema, il testo mostra questa soluzione:

struct A { int v = 0; };
struct B : virtual A {};
struct C : virtual A {};
struct D : B, C {};

Per questa lezione basta conoscere il problema a livello introduttivo. Il punto non è diventare esperti di gerarchie complesse, ma capire perché l'ereditarietà multipla va trattata con prudenza e quasi sempre limitata a casi chiari.

Buone pratiche da portare via

Il testo chiude la teoria con una lista molto sana. Vale la pena lasciarla quasi intatta, perché funziona come checklist mentale di progettazione:

  1. Se usi il polimorfismo, metti un distruttore virtuale nella base.
  2. Usa override su ogni metodo virtuale ridefinito.
  3. Evita lo slicing: preferisci riferimenti, puntatori e smart pointer.
  4. Non usare ereditarietà se la relazione corretta è "ha-un".
  5. Mantieni la classe base semplice e con responsabilità chiare.
  6. Evita gerarchie troppo profonde.
  7. Progetta la base pensando anche alle classi che deriveranno da essa.

Osservazione importante: una base progettata male può rendere fragili tutte le derivate. Questo è uno dei motivi per cui una gerarchia va progettata con molta più disciplina di una semplice collezione di classi scollegate.

Esempio completo: sistema di notifiche

L'obiettivo del testo è molto chiaro: inviare notifiche su canali diversi senza che il codice chiamante conosca i dettagli concreti. Questo è esattamente il tipo di scenario in cui una base astratta e il polimorfismo fanno davvero ordine.

#include <iostream>
#include <memory>
#include <string>
#include <utility>
#include <vector>

class Notifica {
public:
    virtual ~Notifica() = default;
    virtual void invia(const std::string& messaggio) const = 0;
};

class Email : public Notifica {
public:
    explicit Email(std::string destinatario) : destinatario_(std::move(destinatario)) {}

    void invia(const std::string& messaggio) const override {
        std::cout << "[EMAIL a " << destinatario_ << "] " << messaggio << "\n";
    }
private:
    std::string destinatario_;
};

class SMS : public Notifica {
public:
    explicit SMS(std::string numero) : numero_(std::move(numero)) {}

    void invia(const std::string& messaggio) const override {
        std::cout << "[SMS a " << numero_ << "] " << messaggio << "\n";
    }
private:
    std::string numero_;
};

class Push : public Notifica {
public:
    explicit Push(std::string deviceId) : deviceId_(std::move(deviceId)) {}

    void invia(const std::string& messaggio) const override {
        std::cout << "[PUSH a " << deviceId_ << "] " << messaggio << "\n";
    }
private:
    std::string deviceId_;
};

int main() {
    std::vector<std::unique_ptr<Notifica>> canali;
    canali.push_back(std::make_unique<Email>("alice@example.com"));
    canali.push_back(std::make_unique<SMS>("+39 333 1234567"));
    canali.push_back(std::make_unique<Push>("DEVICE-XYZ"));

    for (const auto& c : canali) {
        c->invia("Compito assegnato: esercizi di C++");
    }
}

Che cosa mostra davvero l'esempio delle notifiche

  • main() usa un'interfaccia comune: Notifica.
  • Ogni canale implementa il proprio comportamento concreto.
  • Aggiungere un nuovo canale non richiede modifiche al ciclo principale.

Questo è un caso pulito di polimorfismo con base astratta. Il codice che usa il sistema non ha bisogno di sapere se il messaggio partirà via email, SMS o push. Gli basta sapere che sta lavorando con qualcosa che sa inviare una notifica.

Messaggio chiave: un buon progetto sposta la variabilità dentro le derivate e lascia il codice cliente stabile, semplice e leggibile.

Errori tipici da mostrare in classe

Il testo chiude con quattro errori molto ben scelti, perché sono proprio quelli che producono fraintendimenti ricorrenti.

(A) Metodo non virtuale

Sintomo: chiami un metodo su Base*, ma viene eseguita la versione della base anche quando il puntatore punta a una derivata.

Correzione: rendere il metodo virtual nella base e usare override nella derivata.

(B) Distruttore non virtuale

Sintomo: elimini un oggetto derivato tramite puntatore a base e la distruzione non è corretta.

Correzione: aggiungere virtual ~Base() = default;

(C) Firma sbagliata

Sintomo: pensi di aver fatto override, ma in realtà hai creato un altro metodo.

Correzione: usare override, così il compilatore controlla la firma.

(D) Slicing

Sintomo: copi una derivata in una base per valore e perdi il comportamento specifico.

Correzione: usare riferimenti, puntatori o smart pointer.

Mini-esercizi finali

Il testo della lezione si chiude con quattro mini-esercizi molto mirati. Sono utili per trasformare la teoria in allenamento immediato, senza aspettare il laboratorio finale.

  1. Crea una gerarchia con classe base Dipendente e metodo virtuale stipendioMensile() const. Deriva DipendenteFisso e DipendenteOrario. Poi somma gli stipendi in un vector<unique_ptr<Dipendente>>.
  2. Definisci una classe astratta Figura con area() e perimetro(). Implementa Quadrato, Rettangolo e Cerchio. Stampa le aree con un ciclo polimorfico.
  3. Togli virtual da un metodo che dovrebbe essere polimorfico e osserva che cosa succede. Poi ripristina virtual e aggiungi override.
  4. Prova un caso di name hiding e risolvilo con using Base::nomeMetodo;.

Osservazione didattica: questi esercizi sono piccoli, ma ognuno colpisce uno snodo concettuale importante: base astratta, ciclo polimorfico, ruolo di virtual, ruolo di override, gestione del name hiding.

Appendice: parole chiave da ricordare

  • virtual: abilita il dispatch dinamico
  • override: verifica che il metodo ridefinisca correttamente quello della base
  • final: impedisce ulteriori override o ulteriori derivazioni
  • protected: accessibile alle derivate ma non dall'esterno
  • dynamic_cast: downcast sicuro a runtime
  • classe astratta: contiene almeno un metodo virtuale puro

Il testo chiude con un piccolo esempio su final, utile per vedere sia un metodo finale sia una classe finale.

struct Base {
    virtual void f() {}
};

struct Der : Base {
    void f() override final {}
};

struct Der2 : Der {
    // void f() override {}   // errore: f() è final
};

struct ClasseFinale final : Base {
    void f() override {}
};

// struct Impossibile : ClasseFinale {};  // errore

Perché è utile ricordarlo: final non è centrale come virtual o override, ma completa bene il quadro: puoi impedire sia ulteriori ridefinizioni di un metodo sia ulteriori derivazioni di una classe.

Sintesi finale

  • l'ereditarietà multipla esiste, ma va usata con molta disciplina;
  • il diamond problem mostra perché certe gerarchie possono diventare ambigue;
  • la base deve restare semplice, chiara e progettata pensando alle future derivate;
  • gli esempi migliori di polimorfismo sono quelli in cui il codice cliente non conosce i dettagli concreti;
  • gli errori tipici si ripetono sempre sugli stessi punti: virtual, distruttore, firma, slicing.

Sintesi finale della teoria: l'ereditarietà in C++ è potente, ma diventa davvero utile solo quando la relazione concettuale è corretta, la base è progettata bene e il polimorfismo viene usato per semplificare il codice, non per complicarlo.
