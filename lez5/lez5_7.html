Casting tra base e derivata
Upcast, downcast, dynamic_cast e uso prudente

Perché parlare di cast in una gerarchia

Una volta che hai una gerarchia con base e derivate, può capitarti di voler passare da un tipo all'altro. Non tutti questi passaggi hanno però lo stesso grado di sicurezza.

Il testo distingue bene due casi:

  • upcast: da derivata a base;
  • downcast: da base a derivata.

Questa distinzione è essenziale, perché i due movimenti non sono simmetrici. Salire verso la base è naturale; scendere verso una derivata specifica richiede molta più cautela.

Upcast: passare da derivata a base è sicuro

L'upcast consiste nel trattare un oggetto derivato come un oggetto della sua base. Questo passaggio è sicuro, perché la derivata è davvero anche una base.

Il testo usa questo esempio:

Cane c;
Animale& ra = c;
Animale* pa = &c;

Qui tutto funziona perché un Cane è anche un Animale. Non stiamo inventando una relazione che non c'è: stiamo semplicemente guardando l'oggetto da una prospettiva più generale.

Regola mentale: salire verso la base è coerente con il significato di IS-A.

Downcast: passare da base a derivata è più delicato

Il downcast è l'operazione opposta: hai un puntatore o riferimento a base e vuoi recuperare una derivata specifica. Qui bisogna stare attenti, perché non ogni Base* punta davvero a un oggetto di tipo Derivata.

Questo è il punto cruciale: dal fatto che un oggetto derivato possa essere visto come base non segue automaticamente che ogni base possa essere trattata come una certa derivata. Sarebbe un salto logico ingiustificato.

Quando entra in gioco dynamic_cast

Quando serve controllare il tipo reale a runtime, si può usare dynamic_cast. Questa operazione non "indovina" a caso: verifica se l'oggetto reale appartiene davvero al tipo richiesto.

Il testo propone questo esempio:

#include <iostream>

struct Base {
    virtual ~Base() = default;
};

struct Der1 : Base {
    void x() const { std::cout << "Der1\n"; }
};

struct Der2 : Base {
    void y() const { std::cout << "Der2\n"; }
};

void prova(Base* p) {
    if (auto d1 = dynamic_cast<Der1*>(p)) {
        d1->x();
    } else if (auto d2 = dynamic_cast<Der2*>(p)) {
        d2->y();
    } else {
        std::cout << "Tipo sconosciuto\n";
    }
}

Che cosa osservare nell'esempio

La funzione prova(...) riceve un Base*. Quindi, a livello statico, sa solo che sta lavorando con qualcosa che appartiene alla gerarchia di Base. Non sa ancora se quel qualcosa sia un Der1, un Der2 o altro.

Per questo prova in modo prudente:

  • se il cast a Der1* riesce, usa x();
  • altrimenti prova con Der2*;
  • se anche quello fallisce, gestisce il caso sconosciuto.

Questa è la logica corretta del downcast sicuro: non si assume, si verifica.

Due cose da ricordare su dynamic_cast

  • dynamic_cast richiede una gerarchia polimorfica, cioè una base con almeno un metodo virtuale. Nell'esempio il distruttore virtuale nella base serve anche a questo.
  • se il cast a puntatore fallisce, il risultato è nullptr.

Questo dettaglio è importante perché rende il fallimento verificabile in modo pulito e sicuro. Non ottieni un oggetto finto o un comportamento casuale: ottieni un puntatore nullo, e il programma può reagire di conseguenza.

Perché non bisogna abusare del downcast

Il testo chiude con una regola progettuale molto sana: se puoi ottenere lo stesso risultato con un metodo virtuale ben progettato, quello è di solito l'approccio migliore.

Questa osservazione va presa sul serio. Se un programma passa il tempo a fare cast dalla base alle varie derivate per capire "chi è" l'oggetto, probabilmente la base non sta esprimendo abbastanza bene il contratto comune.

Segnale d'allarme: se devi fare dynamic_cast ovunque, forse il progetto sta chiedendo una interfaccia virtuale migliore, non più cast.

Micro-confronto: quando il cast non serve

Molte volte il bisogno di dynamic_cast scompare se la base espone già il comportamento giusto. In quel caso il codice chiamante non deve più chiedere "che tipo sei?", ma solo usare il contratto comune.

#include <iostream>

struct Base {
    virtual ~Base() = default;
    virtual void descrivi() const = 0;
};

struct Der1 : Base {
    void descrivi() const override { std::cout << "Sono Der1\n"; }
};

struct Der2 : Base {
    void descrivi() const override { std::cout << "Sono Der2\n"; }
};

void mostra(const Base& b) {
    b.descrivi();   // nessun cast
}

Qui il codice ad alto livello non ha bisogno di distinguere manualmente tra Der1 e Der2. Chiede direttamente il comportamento comune, ed è la gerarchia a fare il resto.

Quando dynamic_cast ha senso

Dire che va usato con prudenza non significa che vada demonizzato. Ci sono casi in cui ha senso:

  • quando devi davvero distinguere sottotipi concreti in un punto specifico del programma;
  • quando stai integrando codice esistente e la gerarchia non è stata progettata fin dall'inizio nel modo migliore;
  • quando vuoi fare un controllo runtime esplicito su una gerarchia polimorfica.

Ma la regola generale resta valida: prima prova a modellare il comportamento comune con metodi virtuali; solo dopo valuta se un cast è davvero inevitabile.

Sintesi finale

  • l'upcast da derivata a base è sicuro;
  • il downcast da base a derivata è delicato e va verificato;
  • dynamic_cast richiede una gerarchia polimorfica;
  • se il cast a puntatore fallisce, ottieni nullptr;
  • quando è possibile, un metodo virtuale ben progettato è preferibile a cast frequenti.

Sintesi finale: salire verso la base è il gesto naturale della gerarchia; scendere verso una derivata richiede una verifica, perché il tipo reale potrebbe non essere quello che speri.
