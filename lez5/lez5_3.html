Costruttori, distruttori e ordine di inizializzazione
Che cosa succede davvero quando nasce e muore un oggetto derivato

Perché questa pagina conta davvero

Dopo aver visto la sintassi della derivazione, il passo successivo è capire il ciclo di vita degli oggetti in una gerarchia. Questa parte viene spesso sottovalutata, ma in realtà è uno dei punti che spiegano meglio come C++ ragiona sugli oggetti derivati.

Quando crei un oggetto derivato, non stai creando una cosa "monolitica". Stai creando un oggetto che contiene anche la sua parte di base. Per questo motivo la costruzione e la distruzione seguono un ordine preciso, e quell'ordine non è un dettaglio secondario: determina come si inizializzano i dati e come si rilasciano le risorse.

Idea chiave: prima nasce la base, poi la derivata; quando l'oggetto muore, l'ordine si inverte.

Ordine di costruzione e distruzione

Quando si crea un oggetto derivato accadono due passi fondamentali:

  1. viene costruita prima la parte di base;
  2. poi viene costruita la parte specifica della derivata.

Quando invece l'oggetto viene distrutto, l'ordine è inverso:

  1. si distrugge prima la derivata;
  2. poi la base.

Questo comportamento è logico. La parte derivata si appoggia sulla base, quindi la base deve esistere per prima. Allo stesso modo, quando si distrugge l'oggetto, la parte più specifica va smontata prima di eliminare il supporto comune su cui si regge.

Esempio: vedere l'ordine con i propri occhi

Il modo più semplice per fissare questa regola è osservare un programma che stampa i messaggi dei costruttori e dei distruttori.

#include <iostream>

class Base {
public:
    Base()  { std::cout << "Base()\n"; }
    ~Base() { std::cout << "~Base()\n"; }
};

class Derivata : public Base {
public:
    Derivata()  { std::cout << "Derivata()\n"; }
    ~Derivata() { std::cout << "~Derivata()\n"; }
};

int main() {
    Derivata d;
}

Output atteso

Eseguendo il programma otteniamo:

Base()
Derivata()
~Derivata()
~Base()

Questo output rende evidente la regola:

  • in costruzione: prima Base(), poi Derivata();
  • in distruzione: prima ~Derivata(), poi ~Base().

Non è solo una curiosità. È il comportamento che spiega perché la base deve essere inizializzata correttamente e perché una derivata può contare sul fatto che la parte base esista già quando il suo costruttore inizia a lavorare.

Che cosa significa operativamente

Se la base viene costruita per prima, allora la derivata non può comportarsi come se la base fosse un dettaglio facoltativo. La base è una parte reale dell'oggetto, e il linguaggio garantisce che nasca prima della derivata proprio per rendere coerente l'intera struttura.

Allo stesso modo, quando l'oggetto viene distrutto, la derivata viene smontata prima. Questo è importante soprattutto quando entreranno in gioco risorse, puntatori, file o altre componenti che richiedono una distruzione ordinata e sicura.

Da ricordare: l'ordine non dipende dal compilatore o dall'umore del programma. È parte del modello del linguaggio.

Base con costruttore parametrico

Un altro punto essenziale del testo è questo: se la classe base richiede argomenti, la derivata deve inizializzarla nella lista di inizializzazione. Non può rimandare, non può "sistemare dopo". La base va costruita subito, e va costruita bene.

Questo è il motivo per cui nella lista di inizializzazione del costruttore della derivata compare spesso una chiamata esplicita al costruttore della base.

Esempio: Rettangolo e Parallelepipedo

Il testo propone questo esempio:

class Rettangolo {
public:
    Rettangolo(double w, double h) : w_(w), h_(h) {}
    double area() const { return w_ * h_; }
private:
    double w_, h_;
};

class Parallelepipedo : public Rettangolo {
public:
    Parallelepipedo(double w, double h, double d)
        : Rettangolo(w, h), d_(d) {}

    double volume() const { return area() * d_; }
private:
    double d_;
};

Qui Parallelepipedo non si limita ad aggiungere il campo d_. Inizializza anche la parte base Rettangolo, passandole larghezza e altezza. Solo dopo aggiunge la propria profondità e può definire il metodo volume().

Perché questo esempio è utile

L'esempio serve soprattutto a mostrare due cose:

  • il riuso del costruttore della base;
  • il riuso dei metodi della base, come area().

Dal punto di vista strettamente didattico funziona bene, perché permette di concentrarsi su inizializzazione e riuso senza introdurre troppi dettagli. Il testo, però, fa giustamente anche una nota metodologica importante: in un progetto reale, dal punto di vista concettuale, si potrebbe preferire la composizione.

Questo è un ottimo promemoria: non tutti gli esempi che sono utili a spiegare la meccanica del linguaggio sono automaticamente il miglior modello del dominio reale.

Nota didattica importante

Quando studi il linguaggio, a volte incontri esempi costruiti per isolare un meccanismo preciso. Qui il meccanismo è chiarissimo: la derivata richiama il costruttore della base e riusa i suoi metodi.

Ma progettare bene non significa solo sapere che una cosa si può fare. Significa anche chiedersi se sia il modello più adatto. In questa lezione stiamo quindi lavorando su due piani insieme:

  • piano del linguaggio: come funziona la costruzione di un oggetto derivato;
  • piano della progettazione: quando l'ereditarietà è davvero il modello giusto e quando invece è meglio comporre oggetti.

Sintesi finale

  • quando nasce un oggetto derivato, nasce prima la base e poi la derivata;
  • quando l'oggetto viene distrutto, si smonta prima la derivata e poi la base;
  • se la base richiede argomenti, la derivata deve inizializzarla nella lista di inizializzazione;
  • riusare costruttori e metodi della base è una parte naturale della derivazione;
  • un esempio tecnicamente corretto non è sempre il miglior modello concettuale del dominio.

Sintesi finale: capire l'ordine di costruzione e distruzione significa capire che una derivata non vive sopra la base "per modo di dire": la contiene davvero, e C++ gestisce questa relazione in modo rigoroso.
