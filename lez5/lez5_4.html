Object slicing
Che cosa perdi quando copi una derivata in una base per valore

Che cos'è lo slicing

Lo object slicing si verifica quando un oggetto derivato viene copiato in un oggetto base per valore. In quel momento la parte specifica della derivata viene "tagliata via".

Il nome è molto efficace: immagina davvero di prendere un oggetto più ricco e di tenerne solo il pezzo che corrisponde alla base. Tutto ciò che appartiene solo alla derivata non entra nel nuovo oggetto base.

Punto critico: lo slicing non è un bug del compilatore e non è un comportamento misterioso. È la conseguenza normale del fatto che stai creando un oggetto di tipo Base, non un alias dell'oggetto derivato originale.

Perché è un problema didatticamente importante

Questa è una delle prime situazioni in cui gli studenti vedono chiaramente che "ereditarietà" non significa che tutti i tipi della gerarchia siano intercambiabili in qualunque modo. Trattare una derivata come una base è sicuro solo in certi contesti. Se lo fai per valore, cambi la natura dell'oggetto.

Lo slicing è quindi un punto di svolta concettuale: spinge a capire perché il polimorfismo reale in C++ si appoggia a riferimenti, puntatori e, nel C++ moderno, spesso a smart pointer.

Esempio minimo

Il testo propone questo esempio, volutamente piccolo:

#include <iostream>

class Base {
public:
    void f() const { std::cout << "Base::f\n"; }
};

class Derivata : public Base {
public:
    void g() const { std::cout << "Derivata::g\n"; }
};

int main() {
    Derivata d;
    Base b = d;   // slicing
    b.f();
    // b.g();     // errore: b è un oggetto Base
}

Il cuore dell'esempio sta tutto nella riga:

Base b = d;

Questa riga non "ricorda" che d era una Derivata. Sta creando un nuovo oggetto di tipo Base.

Che cosa succede davvero?

Il punto più importante del testo è questo: b non è un alias di d. b è un nuovo oggetto di tipo Base, ottenuto copiando soltanto la parte Base di d.

Questo significa che tutto ciò che appartiene solo alla derivata non entra nel nuovo oggetto. Nel nostro esempio, il metodo g() non può essere chiamato su b per una ragione molto semplice: b non è una Derivata. È davvero e soltanto un oggetto Base.

Formula mentale corretta: non pensare "ho una derivata vista come base". Qui devi pensare: "ho costruito una nuova base copiando solo la parte di base della derivata".

Perché il codice compila ma il significato cambia

Il codice compila perché una derivata contiene anche una parte base, quindi il compilatore sa come costruire un oggetto Base a partire da quella porzione. Dal punto di vista del linguaggio non c'è nulla di scorretto.

Il problema è semantico e progettuale: se il tuo obiettivo era preservare l'identità completa dell'oggetto derivato, allora la copia per valore nella base non fa quello che pensavi. Ti lascia in mano un oggetto più povero, con meno comportamento e meno informazione.

Come si evita lo slicing

Il testo indica tre strade fondamentali:

  • usare riferimenti a base (Base&);
  • usare puntatori a base (Base*);
  • usare smart pointer quando serve gestione dinamica della memoria.

Queste tre soluzioni hanno in comune una cosa: non stanno creando una nuova Base per valore. Stanno mantenendo un legame con l'oggetto reale, che può continuare a essere una Derivata.

Riferimento a base: nessuna copia, nessun taglio

Se usi un riferimento, non stai creando un nuovo oggetto. Stai dando un altro nome allo stesso oggetto già esistente.

Derivata d;
Base& r = d;

Qui non avviene slicing, perché r non è un oggetto Base autonomo: è un riferimento che si appoggia a d. Questa distinzione è essenziale e prepara direttamente il terreno al polimorfismo che vedremo nella prossima pagina.

Puntatore a base: stesso principio

Anche con i puntatori il principio è lo stesso:

Derivata d;
Base* p = &d;

Nemmeno qui nasce una nuova Base. Il puntatore punta all'oggetto esistente. Di conseguenza, il programma può ancora lavorare sullo stesso oggetto reale, senza perdere la sua parte derivata.

Naturalmente, il semplice fatto di usare un puntatore non basta ancora a ottenere polimorfismo dinamico: per quello ci serviranno i metodi virtual. Ma intanto abbiamo evitato il taglio dell'oggetto.

Smart pointer e C++ moderno

Quando la gestione degli oggetti richiede allocazione dinamica, nel C++ moderno non conviene pensare subito a new e delete. Conviene pensare a smart pointer come std::unique_ptr e, più raramente, std::shared_ptr.

In questo contesto, gli smart pointer sono utili per due motivi:

  • evitano lo slicing, perché non copiano l'oggetto derivato in una base per valore;
  • gestiscono in modo più sicuro la memoria e il ciclo di vita degli oggetti.

Regola pratica: se vuoi polimorfismo e gestione sicura della memoria, std::unique_ptr<Base> è spesso il punto di partenza migliore.

Sintesi finale

  • lo slicing avviene quando una derivata viene copiata in una base per valore;
  • in quel caso la parte specifica della derivata viene persa;
  • l'oggetto base ottenuto non è un alias dell'oggetto derivato originale;
  • per evitare slicing si usano riferimenti, puntatori e smart pointer;
  • questa è una delle ragioni per cui il polimorfismo in C++ non si costruisce su copie per valore.

Sintesi finale: se copi una derivata in una base per valore, non stai "conservando il polimorfismo". Stai costruendo un nuovo oggetto base, più povero dell'originale.
