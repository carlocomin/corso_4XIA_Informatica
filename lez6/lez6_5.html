Le visite nel problem solving algoritmico

Obiettivi della pagina

Fin qui abbiamo studiato le visite come meccanismi di attraversamento: preordine, inordine, postordine, DFS e BFS. Ora dobbiamo compiere il salto davvero importante: capire che una visita non è un esercizio formale, ma un modo sistematico di risolvere problemi su una struttura gerarchica.

Questa è la pagina in cui le visite smettono di essere "ordini di stampa" e diventano tecniche per contare, cercare, copiare, valutare, distruggere, misurare e prendere decisioni algoritmiche. È anche la pagina che prepara bene l'arrivo dei BST: prima ancora di specializzare gli alberi, vogliamo saperli usare.

Idea chiave: scegliere una visita significa scegliere un modo di scomporre il problema in sottoproblemi più piccoli e di combinare i risultati in modo coerente.

La domanda giusta non è "come visito?", ma "che cosa devo ottenere?"

Quando uno studente incontra le visite per la prima volta, spesso pensa così: "dimmi se devo usare preordine, inordine o postordine". In realtà questo modo di ragionare è troppo meccanico. La domanda corretta è un'altra: qual è la natura del risultato che voglio ottenere?

Per esempio:

  • se voglio elaborare il nodo prima dei figli, il preordine è naturale;
  • se il nodo dipende dai risultati dei figli, il postordine è spesso la scelta giusta;
  • se mi interessa l'ordine sinistra-nodo-destra, l'inordine è il candidato naturale;
  • se conta la distanza dalla radice, BFS diventa molto interessante.

In altre parole, la visita non si sceglie a memoria. Si sceglie perché è coerente con la forma del problema.

Primo schema mentale: problema locale + sottoproblemi

Molti algoritmi sugli alberi hanno una struttura ricorrente:

  1. risolvo il caso base dell'albero vuoto;
  2. ricavo informazioni dal sottoalbero sinistro;
  3. ricavo informazioni dal sottoalbero destro;
  4. combino questi risultati con il nodo corrente.

Questo schema è uno dei motivi per cui gli alberi sono didatticamente così importanti. Rendono concreta l'idea di decomporre un problema in sottoproblemi simili, trattarli ricorsivamente e poi ricombinare il tutto.

Osservazione utile: anche quando il codice finale sembra molto diverso, sotto spesso resta questo stesso schema logico.

Problema 1: contare nodi e foglie

Un primo esempio molto naturale è il conteggio. Se voglio sapere quanti nodi ha un albero, il ragionamento è semplice: il numero totale è dato da 1 per il nodo corrente, più i nodi del sottoalbero sinistro, più i nodi del sottoalbero destro.

int countNodes(const Node* p) {
    if (p == nullptr) return 0;
    return 1 + countNodes(p->left) + countNodes(p->right);
}

Per contare le foglie cambia solo il criterio locale: un nodo è una foglia se non ha né figlio sinistro né figlio destro.

int countLeaves(const Node* p) {
    if (p == nullptr) return 0;
    if (p->left == nullptr && p->right == nullptr) return 1;
    return countLeaves(p->left) + countLeaves(p->right);
}

Qui non stiamo semplicemente "scorrendo" l'albero. Stiamo usando la sua struttura per costruire una formula ricorsiva naturale.

Problema 2: calcolare l'altezza

L'altezza è un altro esempio didatticamente perfetto, perché costringe a combinare i risultati dei due sottoalberi. L'idea è: l'altezza di un nodo è 1 più il massimo tra l'altezza del sottoalbero sinistro e quella del sottoalbero destro.

#include <algorithm>

int height(const Node* p) {
    if (p == nullptr) return 0;
    int hs = height(p->left);
    int hd = height(p->right);
    return 1 + std::max(hs, hd);
}

Questo è un tipico caso in cui il nodo corrente dipende dai risultati ottenuti dai figli. Per questo il ragionamento è concettualmente vicino al postordine: prima ricavo le informazioni dei sottoalberi, poi costruisco il risultato del nodo corrente.

Problema 3: cercare un valore in un albero generico

Prima di arrivare ai BST, immaginiamo un albero binario generico, cioè senza nessuna proprietà di ordinamento. In questo caso, cercare un valore significa esplorare la struttura finché il valore non viene trovato oppure finché l'albero finisce.

bool contains(const Node* p, int x) {
    if (p == nullptr) return false;
    if (p->key == x) return true;
    return contains(p->left, x) || contains(p->right, x);
}

Questa funzione è molto istruttiva: mostra che, in assenza di una proprietà speciale, l'albero va comunque esplorato in modo sistematico. Solo più avanti, con i BST, vedremo che l'ordinamento permette di evitare rami inutili e quindi di cercare in modo più efficiente.

Problema 4: copiare l'albero

Copiare un albero non significa copiare solo il valore della radice. Significa ricostruire tutta la struttura con nuovi nodi, mantenendo la stessa forma e gli stessi dati. Questo è un caso in cui il preordine è molto intuitivo: creo prima il nodo corrente, poi gli collego le copie dei due sottoalberi.

Node* cloneTree(const Node* p) {
    if (p == nullptr) return nullptr;

    Node* q = new Node{p->key, nullptr, nullptr};
    q->left = cloneTree(p->left);
    q->right = cloneTree(p->right);
    return q;
}

Questa funzione è un ottimo esempio di problem solving su alberi: l'algoritmo è compatto, ma dietro c'è un'idea forte di corrispondenza struttura per struttura.

Problema 5: distruggere correttamente l'albero

Con strutture dinamiche non basta "smettere di usarle": bisogna liberare la memoria. Qui il postordine è il modello più naturale. Prima elimino i sottoalberi, poi elimino il nodo corrente. Se facessi il contrario, perderei i collegamenti ai figli e quindi non potrei più liberarli correttamente.

void destroyTree(Node* p) {
    if (p == nullptr) return;
    destroyTree(p->left);
    destroyTree(p->right);
    delete p;
}

Questo è uno degli esempi più belli perché fa vedere che il significato della visita non è affatto arbitrario. Il postordine qui non è "una delle tre visite possibili": è quella coerente con la dipendenza logica del problema.

Problema 6: valutare un albero di espressione

Negli alberi di espressione, i nodi interni contengono operatori e le foglie contengono valori o simboli. Se voglio valutare l'espressione, devo prima conoscere il valore dei due sottoalberi e solo dopo applicare l'operatore del nodo corrente. Anche qui il modello naturale è quello del postordine.

        *
      /   \
     +     5
    / \
   2   3

La logica è:

  • valuto il sottoalbero sinistro: 2 + 3 = 5;
  • valuto il sottoalbero destro: 5;
  • applico l'operatore in radice: 5 * 5 = 25.

Questo esempio è molto formativo perché fa vedere come l'ordine della visita rifletta l'ordine logico della dipendenza tra operazioni.

Problema 7: stampare o ricostruire un ordine significativo

Le visite servono anche a produrre rappresentazioni diverse della stessa struttura. In un albero di espressione:

  • il preordine si lega naturalmente alla forma prefissa;
  • l'inordine alla forma infissa;
  • il postordine alla forma postfissa.

Questo è un punto importante: una visita può essere usata per "estrarre" da una struttura un ordine utile, leggibile o computazionalmente significativo. Più avanti, nei BST, vedremo che l'inordine produce addirittura i dati già ordinati.

Problema 8: BFS quando conta la vicinanza alla radice

Non tutti i problemi si risolvono bene con una DFS. Se voglio trovare il primo nodo che soddisfa una proprietà al livello minimo possibile, BFS diventa molto più naturale. Supponi, per esempio, di voler cercare il primo valore pari il più vicino possibile alla radice. Una DFS potrebbe scendere troppo in profondità in un ramo e trovare una soluzione "lontana" prima di controllare nodi più vicini in altri rami.

Con una BFS, invece, esamino prima tutti i nodi del livello 0, poi quelli del livello 1, poi quelli del livello 2. Questo garantisce che la prima soluzione trovata sia tra le più vicine alla radice.

Regola pratica: quando il problema parla di livelli, distanza minima o nodo più vicino alla radice, la BFS è spesso la prima strategia da prendere in considerazione.

Una mappa ragionata: quale visita per quale problema?

Problema Visita o strategia naturale Perché
Copiare l'albero Preordine Creo il nodo prima di costruire i figli
Valutare un'espressione Postordine Il nodo dipende dai risultati dei figli
Distruggere l'albero Postordine Prima libero i figli, poi il nodo
Produrre forma infissa Inordine La posizione sinistra-nodo-destra è significativa
Trovare il nodo più vicino alla radice con una proprietà BFS La visita procede per livelli

Questa tabella non va letta come un formulario rigido, ma come una guida al ragionamento. Il punto è sempre capire la relazione tra il nodo corrente e i suoi sottoalberi.

Importanza didattica: le visite insegnano un metodo

Uno degli aspetti più forti degli alberi, dal punto di vista formativo, è che costringono a ragionare in termini di struttura, caso base, sottoproblemi e combinazione dei risultati. Questo è molto più importante della singola sintassi del linguaggio.

Se uno studente impara davvero a usare le visite, impara anche un modo di affrontare i problemi algoritmici:

  • identificare la struttura del problema;
  • trovare il caso base;
  • scomporre in sottoproblemi omogenei;
  • scegliere l'ordine di elaborazione più coerente;
  • ricombinare i risultati in modo corretto.

Per questo gli alberi non sono solo una struttura dati del programma di C/C++. Sono anche un ottimo laboratorio di pensiero algoritmico.

Punti chiave da fissare

Se questa lezione è stata capita bene, dovresti avere chiaro che:

  • le visite servono a risolvere problemi, non solo a stampare nodi;
  • la scelta della visita dipende dal tipo di risultato cercato;
  • molti problemi sugli alberi si esprimono come combinazione dei risultati dei sottoalberi;
  • DFS e BFS diventano strumenti diversi per obiettivi diversi;
  • gli alberi allenano a pensare in termini di decomposizione ricorsiva del problema.

Questo è il punto in cui le visite diventano davvero utili. Adesso siamo pronti a specializzare il discorso sugli alberi binari di ricerca, che aggiungono una proprietà di ordinamento e aprono la strada a ricerca, inserimento e cancellazione efficienti.

Sintesi finale. Le visite sono strumenti di problem solving perché permettono di trasformare un problema su un albero in una sequenza ordinata di sottoproblemi sui suoi sottoalberi. Preordine, inordine, postordine e BFS non sono formule da ricordare a memoria: sono scelte algoritmiche motivate dal tipo di risultato che vogliamo ottenere. Nella Lez6_6 vedremo come questa idea si potenzia ulteriormente negli alberi binari di ricerca.
